# SuperCtx v0.2 Hub-and-Shim Architecture

**Status**: Proposed for v0.2 direction
**Date**: 2026-06-02

## Context

In SuperCtx v0.1, the synchronization model is **inward-sync**:
```text
tool instruction files (sources) -> .ctx/SUPERCTX.md (generated aggregation)
```
While this successfully proved the MVP and enabled cross-tool context aggregation, it presents a major UX drawback: agents and users must edit tool-specific files and then manually run `/superctx:sync`. This workflow is error-prone, prone to drift, and asks the developer to hand-manage files that should be handled automatically by the context engine.

In SuperCtx v0.2, the flow is **inverted** to a **hub-and-shim** architecture:
```text
.ctx/SUPERCTX.md (canonical source) -> tool instruction files (generated shims)
```
This document defines the architecture, contract rules, and command specifications for this v0.2 model.

## Architectural Components

The v0.2 architecture divides repository context files into three strict roles:

### 1. The Canonical Hub (`.ctx/SUPERCTX.md`)
- **Role**: The single source of truth for the project's active instruction context in v0.2.
- **Editing**: Users and agents update `.ctx/SUPERCTX.md` directly. It is a hand-edited, version-controlled Markdown file.
- **Structure**: A centralized structured aggregation of instructions, organized by section or source provenance, serving as the universal reference for all coding tools in the workspace.

### 2. The Shim Contract (Registered Tool Files)
- **Role**: Tool-specific files (e.g. `.claude/CLAUDE.md`, `.codex/AGENTS.md`) act as thin adapters/shims mapping the canonical hub to their respective tools.
- **Constraints**:
  - **Generated**: Shims are fully generated by SuperCtx commands and must not be hand-edited in the happy path.
  - **Warning Header**: Every generated shim must contain a clear, human-readable warning header explaining that it is generated by SuperCtx and that manual edits will be overwritten.
  - **Tool-Specific**: Shims are formatted specifically to satisfy the tool they target, serving as a compatibility layer.
  - **Referential**: Shims must explicitly reference or point to `.ctx/SUPERCTX.md` as the canonical source.
  - **Repairable**: If a shim is deleted or modified, SuperCtx commands must be able to detect this drift and cleanly regenerate it.

### 3. Backups (`.ctx/sources/`)
- **Role**: Backup storage for original instruction files created during the initial migration (e.g., when migrating from pre-SuperCtx to v0.2).
- **Status**: Completely **inactive**.
- **Constraint**: Backups must **never** be imported or read as active context by agents. They are for recovery/repair only. Backups can be stale, superseded, or partial recovery copies, so importing them can reintroduce outdated instructions.

## Target v0.2 Command Semantics

In SuperCtx v0.2, the CLI commands change their execution semantics to support the outward-facing hub-and-shim model:

### 1. `init`
- **Behavior**: Scaffolds the `.ctx/` directory if missing. Auto-detects existing instruction files and copies them to `.ctx/sources/` as inactive backups.
- **Hub Construction**: Combines the detected source files into the initial `.ctx/SUPERCTX.md` hub.
- **Migration**: Registers the original paths in `manifest.toml` and migrates the registered paths to generated thin shims pointing to the hub.

### 2. `add <path>`
- **Behavior**: Begins tracking a new instruction file path.
- **Preservation & Migration**: Preserves the original file content in `.ctx/sources/`, incorporates its current content into `.ctx/SUPERCTX.md` (with proper provenance headers), adds the file to `manifest.toml`, and migrates the registered path to a generated shim pointing to the hub.

### 3. `status`
- **Behavior**: Performs health and integrity checks.
- **Verifications**:
  1. All tracked paths in the manifest exist as shims.
  2. Shims have not been modified manually (i.e. they match their expected generated shim format and have not drifted).
  3. The `.ctx/SUPERCTX.md` hub has no structure or mapping errors.
  4. Surfaces any untracked conventions or candidates.

### 4. `sync`
- **Behavior**: Transitioned to be **repair/recovery-oriented** rather than the primary workflow.
- **Shim Regeneration**: If `status` detects that a shim has drifted (been manually modified) or is missing, `sync` is run to rebuild or repair the shims from the canonical `.ctx/SUPERCTX.md` hub. It must not rebuild or overwrite `.ctx/SUPERCTX.md` from shim contents.

## Core Product Rules

1. **One Canonical Hub**: The repository has exactly one active canonical hub (`.ctx/SUPERCTX.md`).
2. **Thin Adapters**: Tool-specific files are generated thin shims that direct agents to look at the hub.
3. **No Active Backups**: Files inside `.ctx/sources/` are strictly legacy backups for safety and must never be loaded by agents as active rules.
4. **Agent-Native Happy Path**: The happy path avoids manual editing of generated shims and `manifest.toml`. In v0.2, `.ctx/SUPERCTX.md` is the sanctioned editable Markdown surface. Agents and users update `.ctx/SUPERCTX.md` directly or use SuperCtx commands.
